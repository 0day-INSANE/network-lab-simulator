❓ Network Lab Help

🎯 Introduction

Network Lab is a professional network simulator that lets you create complex topologies and test network traffic in real time.

🚀 Getting Started

  1. Add devices: Click a device in the sidebar, then click on the canvas to place it
  2. Connect devices: Click "Connect", then select two devices
  3. Configure: Click "Analyze" and then click a device to configure it
  4. Test: Create a packet and send it to see network behavior

📦 Creating Packets

You can create custom packets by specifying:

  • Source: Device sending the packet
  • Destination: Destination IP
  • Ports: Source and destination port
  • Protocol: TCP, UDP or ICMP
  • Flags: TCP Flags (SYN, ACK, FIN, RST, PSH)

🔥 Firewall Configuration

Inbound Rules: Control INCOMING traffic to the device

Outbound Rules: Control OUTGOING traffic from the device

Example JSON rule:

{
  "name": "Allow HTTPS",
  "protocol": "TCP",
  "srcIP": "any",
  "srcPort": "any",
  "dstIP": "any",
  "dstPort": "443",
  "action": "allow",
  "flags": []
}

🧠 Custom Logic

Each device can have custom logic rules. Example:

{
  "name": "Block SSH",
  "field": "destPort",
  "operator": "equals",
  "value": 22,
  "action": "block"
}

Available operators: equals, not_equals, contains, greater_than, less_than, in_list, has_flag

🎨 Colored Connections

🟢 Bright green with animated packets: ACTIVE and ALLOWED communication

🟠 Static orange: Physical connection present but communication INACTIVE or BLOCKED

💾 Saving Projects

Save: Saves to local storage (right-click to see list)

Export TXT Diagram: Generates a text file with all network info

Export Full JSON: Downloads a JSON file with EVERYTHING (devices, connections, configurations, custom types)

Import JSON: Load an exported JSON file to restore everything

⚙️ Keywords & JSON Syntax Guide

📋 FIREWALL RULES (Inbound/Outbound)

Firewall rules control incoming (Inbound) and outgoing (Outbound) traffic.

Basic Structure:

[
  {
    "name": "Rule name",
    "protocol": "TCP|UDP|ICMP|any",
    "srcIP": "source IP or 'any' or subnet (192.168.1.0/24)",
    "srcPort": "source port or 'any'",
    "dstIP": "destination IP or 'any' or subnet",
    "dstPort": "destination port or 'any'",
    "action": "allow|deny",
    "flags": ["SYN", "ACK", "FIN", "RST", "PSH"]
  }
]

📝 Keywords Firewall Rules:

Field Possible Values Description
name Any string Descriptive rule name
protocol "TCP", "UDP", "ICMP", "any" Transport protocol
srcIP "192.168.1.100", "192.168.1.0/24", "any" Source IP (single, subnet, or any)
srcPort "80", "1024-65535", "any" Source port
dstIP "10.0.0.1", "10.0.0.0/8", "any" Destination IP
dstPort "443", "8000-9000", "any" Destination port
action "allow", "deny" Action to execute
flags ["SYN"], ["SYN", "ACK"], [] Required TCP flags (optional)

🎯 Firewall Rule Examples:

// Allow HTTP from anywhere
{
  "name": "Allow HTTP",
  "protocol": "TCP",
  "srcIP": "any",
  "srcPort": "any",
  "dstIP": "any",
  "dstPort": "80",
  "action": "allow",
  "flags": []
}

// Block SSH from the internet
{
  "name": "Block SSH from internet",
  "protocol": "TCP",
  "srcIP": "0.0.0.0/0",
  "srcPort": "any",
  "dstIP": "192.168.1.0/24",
  "dstPort": "22",
  "action": "deny",
  "flags": ["SYN"]
}

// Allow only from admin
{
  "name": "Allow from admin only",
  "protocol": "any",
  "srcIP": "192.168.1.5",
  "srcPort": "any",
  "dstIP": "192.168.1.50",
  "dstPort": "any",
  "action": "allow",
  "flags": []
}

🧠 CUSTOM LOGIC (Advanced)

Custom logic rules are more powerful than firewall rules. They are applied FIRST and use advanced operators.

Basic Structure:

[
  {
    "name": "Logic name",
    "field": "sourceIP|sourcePort|destIP|destPort|protocol|flags",
    "operator": "equals|not_equals|contains|greater_than|less_than|in_list|has_flag",
    "value": "value or array",
    "action": "allow|block"
  }
]

📝 Keywords Logic Rules:

Field Possible Values Description
name Any string Logic name
field "sourceIP", "sourcePort", "destIP", "destPort", "protocol", "flags" Packet field to evaluate
operator "equals", "not_equals", "contains", "greater_than", "less_than", "in_list", "has_flag" Comparison operator
value Number, string, or array [80, 443, 22] Value to compare against
action "allow", "block" Action to execute

🔧 Available Operators:

Operator Usage Example
equals Field == Value "destPort" equals 80
not_equals Field != Value "protocol" not_equals "ICMP"
contains Field contains String "sourceIP" contains "192.168"
greater_than Field > Value "destPort" greater_than 1024
less_than Field < Value "sourcePort" less_than 1024
in_list Field in Array "destPort" in_list [80,443,22]
has_flag Flags contains Flag "flags" has_flag "SYN"

🎯 Logic Rule Examples:

// Block Telnet (port 23)
{
  "name": "Block Telnet",
  "field": "destPort",
  "operator": "equals",
  "value": 23,
  "action": "block"
}

// Web ports only
{
  "name": "Allow only web ports",
  "field": "destPort",
  "operator": "in_list",
  "value": [80, 443, 8080],
  "action": "allow"
}

// Block high ports (possible malware)
{
  "name": "Block high ports",
  "field": "destPort",
  "operator": "greater_than",
  "value": 50000,
  "action": "block"
}

// Internal network only
{
  "name": "Only internal network",
  "field": "sourceIP",
  "operator": "contains",
  "value": "192.168.1",
  "action": "allow"
}

// SYN flood protection
{
  "name": "Block SYN flood",
  "field": "flags",
  "operator": "has_flag",
  "value": "SYN",
  "action": "block"
}

📊 EVALUATION ORDER

When a packet passes through a device, rules are applied in this order:

  1. Custom Logic - If blocking, the packet is dropped immediately
  2. Specific Firewall Rules - Evaluated top to bottom
  3. Default Policy - If no rule matches, the default policy (allow/deny) is applied
  4. Open Port Check - Only for the final recipient, checks if the port is in openPorts[]

💡 BEST PRACTICES

  • Default Deny: Set default to "deny" and allow only what is needed
  • Rule Order: More specific rules should come FIRST
  • Documentation: Use descriptive names for every rule
  • Testing: Always test with the Console Log open
  • Custom Logic: Use for complex patterns or multiple conditions
  • Firewall Rules: Use for exact IP/Port matches

⚠️ IMPORTANT

All devices (base and custom) ALWAYS have available:

  • inboundRules - Array of inbound traffic rules
  • outboundRules - Array of outbound traffic rules
  • logicRules - Array of advanced custom logic rules

These fields are automatically added to ALL devices, including custom ones you create!
